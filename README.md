# 选题：图书信息管理系统
# 选题说明：现拟一个图书信息管理系统，其功能为：添加、减少书籍，以及查询（分为具体查询、分类查询和按时间查询）。同时加入管理员和顾客两种用户，其中管理员能对图书信息进行修改，用户能改购买图书
# JSONL文件生成方法：每次与kilo智能体交互时将自动与github同步并将标准格式数据添加到JSONL文件中（详见AGENYS.md和sync_to_github.ps1）
# 提示词生成方法
1、现在用c++建立一个图书信息库，要求有书名，类型，上架时间，在库数量，图书编号，作者，价格信息
理由：先建立图书信息存储室，存储图书的基本信息
2、给代码文件生成注释
理由：没注释下的复杂代码难以确认问题
3、添加查重判断，如果图书信息（价格、作者、出版日期、书名）完全相同，则将它们合并
理由：在实现时发现出现了多种同样书籍的情况
4、现在添加一些顾客用户，顾客用户可以对图书进行购买、查询操作，顾客可以往自己的账号充值，当顾客账号余额不足以购买书籍时，会提示顾客余额不足，当顾客购买书籍后，对应书籍数量减少
理由：有了图书，就要给这个系统一些使用功能
5、现在加入管理员用户，可以对图书进行增删查询工作，同时只有管理员用户可以注册顾客用户。管理员用户是已知且固定的，无法被更改。
理由：一定要有一名管理员来管理顾客和图书信息
6、顾客用户也需要有密码。当对顾客、图书信息进行更改时，永久影响对应的txt文件。
理由：发现顾客用户不需密码直接登入，这是严重隐患，同时使在程序运行时进行的操作永久受影响
7、添加按类别和按时间段搜索图书功能
理由：区分具体查询和大致查询
8、顾客有对应的账号和昵称，账号必须为字母和数字的集合体，昵称则可出现中文等字符，顾客登陆时需使用账号和密码
理由：常见的登录用户名经常仅使用数字和字母组成防止中文容易造成的问题，而保留昵称则能更好确认用户对应的顾客
9、现在更改书本编号定义方式，每种书本类型对应一种字母（即编号首字母），后面的数字字符代表这是这种类型的书本的第N本入库书本，编号不用手动录入，当有新书本入库时使其自动获得编号。
理由：方便添加图书时的编号操作和分类操作
10、编号首字母是按书本类型分别，例如，我有两本文学类书本，则它们的编号首字母都为W，若我新增一本科幻类书本，则它需要新的字母作为编号首位。
理由：加强按类编号的严谨性
11、每当进行完一次操作时，例如添加图书时将所需要的信息全部输入完后，对现展示出的信息清空再输出
理由：发现每次操作后原有的信息会保留，使程序运行窗口变得臃肿，故得让其简洁化
12、将搜索/类别搜索/时间段搜索整合进查看所有图书，使之可连续筛选，管理员菜单移除独立搜索项
理由：将大致查询分出去显得系统太臃肿了，故将这些搜寻合并
13、进行\"修改图书信息\"操作时，若无任何输入，则认为无变化，不弹出错误提示。
理由：不进行操作时出现崩溃问题，进行修正
14、在顾客界面进行充值、购买图书时，不需要重新输入账号，而是记录当前登录的账号，确认再次输入的密码是否正确即可
理由：减少重复输入用户名造成的臃肿问题
